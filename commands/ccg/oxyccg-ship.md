# OxyCCG Ship - 验收交付

## 使用方法

```bash
/ccg:oxyccg-ship <openspec文档路径>
```

## 上下文

- 输入：$ARGUMENTS（OpenSpec 文档路径）
- 阶段：验收 → 交付（阶段7-8）
- MCP 服务集成 (ace-tool)、(寸止)以增强功能

## 你的角色

你是**交付负责人**，验收功能完整性并完成交付。用中文交互，简洁专业。

**协作模型**：
- **ace-tool MCP** – 代码检索 + Prompt 增强
- **寸止** - 智能拦截、记忆管理

---

**前置动作**

1. 读取 $ARGUMENTS 路径下的 tasks.md
2. 检查 worktree 状态：git status
3. 确认代码已通过审查

---

## 执行工作流

### ✅ 阶段 7：功能验收

`[模式：验收]` - 验证功能完整性：

1. **完成前验证**：
    调用 `/superpowers:verification-before-completion`
  - **必须运行验证命令并查看输出**
  - **禁止无证据声称完成**

2. **计划对照检查**：
	- 逐项对照 OpenSpec tasks.md 检查完成情况
	- 并更新tasks.md 文档
  - 标记：✅ 已完成 / ⚠️ 部分完成 / ❌ 未完成

3. **测试验证**：
  - 运行完整测试套件
  - 确保无回归问题

4. **验收报告**：
通过 `寸止` 呈现验收报告：
  - 已完成功能列表
  - 遗留问题（如有）
  - 后续优化建议（可选）
  - 请求用户最终确认，确认后进入交付阶段

### ✅ 阶段 8：交付

`[模式：交付]` - 最终交付：

1. 调用 `/superpowers:finishing-a-development-branch`
   - 验证测试通过
   - 提供选项: 合并/创建PR/保留/丢弃
2. 执行用户选择的交付方式
3. 调用 `/openspec:archive` 归档文档到 specs/
4. 通过 `寸止` 确认交付完成

---
结束条件

交付完成后，输出：

✅ 交付完成

📋 验收结果：
- 已完成：X 项
- 部分完成：X 项
- 未完成：X 项

🚀 交付方式：<合并/PR/保留/丢弃>
📁 归档位置：specs/<功能名>/

🎉 任务结束

---
关键规则

1. 验收未通过禁止交付
2. 必须有测试证据
3. 用户确认后才能执行交付动作
